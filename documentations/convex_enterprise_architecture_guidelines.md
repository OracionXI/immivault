# Convex Enterprise Backend Architecture Guidelines

This document establishes the guidelines, rules, and best practices for structuring an enterprise-level backend architecture using Convex. Following these patterns ensures scalability, maintainability, type safety, and security.

---

## 1. Core Architecture Principles

Convex provides three primary types of functions. Understanding when and where to use them is the foundation of a scalable architecture:

- **Queries (`query`, `internalQuery`)**:
  - Always read-only, deterministic, and side-effect free.
  - Returns data to the client efficiently with automatic reactivity.
  - Do NOT mutate state or call external services.
- **Mutations (`mutation`, `internalMutation`)**:
  - Used for state changes (writing to the database).
  - Transactional and atomic (all database writes hit or get rolled back).
  - Use these for core transactional logic.
- **Actions (`action`, `internalAction`)**:
  - Used for side effects, external API calls, AI/LLM integration, or long-running heavy compute.
  - Cannot access the database directly; must invoke `runQuery` or `runMutation` to read or write data.

**Rule of Thumb:**
Keep Mutations as fast and atomic as possible to avoid lock contention. Offload heavy computation or 3rd-party API calls to Actions.

## 3. Security & Authorization

Security should not be an afterthought. Every publicly exposed query, mutation, or action must validate the caller.

### 3.1 Custom Middleware / Wrappers
Never implement raw `mutation` or `query` for protected routes. Use a higher-order wrapper pattern (like `convex-helpers` custom functions) to enforce authentication and authorization at the edge.

```typescript
// convex/lib/auth.ts
import { customMutation, customQuery } from "convex-helpers/server/customFunctions";
import { mutation, query } from "../_generated/server";
import { ConvexError } from "convex/values";

// Wrapper that ensures the user is authenticated and injects the user ID
export const authenticatedQuery = customQuery(
  query,
  customCtx(async (ctx) => {
    const identity = await ctx.auth.getUserIdentity();
    if (!identity) throw new ConvexError("Unauthenticated");
    
    // Fetch and inject custom user profile from DB if needed
    const user = await ctx.db
      .query("users")
      .withIndex("by_token", (q) => q.eq("tokenIdentifier", identity.tokenIdentifier))
      .unique();
      
    if (!user) throw new ConvexError("User not found in database");
    
    return { user }; // Merges `user` into `ctx` for downstream functions
  })
);

// convex/users/queries.ts
export const getProfile = authenticatedQuery({
  args: {},
  handler: async (ctx) => {
    // Boilerplate-free! We already know `ctx.user` is populated and authenticated.
    return ctx.user; 
  }
});
```

### 3.2 Internal Functions
If a function should **only** be triggered by another Convex function (like an Action, Cron, or HTTP webhook), use `internalQuery`, `internalMutation`, or `internalAction`. 
**Rule:** Never expose critical automated operations on the public API endpoints.

## 4. Schema Design, Indexing & Typing

A well-structured and typed database guarantees longevity. 

### 4.1 Schema Definition
All tables must be declared in `schema.ts`. Avoid overly nested objects. Favor relational IDs (`v.id("tableName")`) to mimic highly normalized relational integrity if required.

### 4.2 Indexing is Mandatory for Scaling
Do not use `ctx.db.query("...").filter(...)` for production queries over large datasets, as it results in full table scans. Use `.withIndex(...)` heavily.

- **Primary Indexes:** Single field indexes for simple lookups (e.g., `by_userId`).
- **Compound Indexes:** Use them when filtering by multiple equality conditions, or sorting on one condition.
- **Search Indexes:** Use `searchIndex()` for robust autocomplete or text search capabilities.
- **Vector Indexes:** Use `vectorIndex()` for semantic search and AI retrieval (RAG).

```typescript
// convex/schema.ts
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({
  bookings: defineTable({
    userId: v.id("users"),
    status: v.union(v.literal("pending"), v.literal("confirmed"), v.literal("cancelled")),
    appointmentDate: v.number(), 
  })
    .index("by_user", ["userId"])
    .index("by_status", ["status"])
    .index("by_user_and_status", ["userId", "status"]) // Compound index for frequent querying
});
```

## 5. Error Handling

Return explicit application errors rather than native TypeScript `Error` to ensure the client can capture and display them properly.

- Use **`ConvexError`** for all expected logic violations (e.g., "Insufficient balance", "Permission denied"). `ConvexError` payloads are serialized and sent to the client.
- Avoid throwing raw strings or standard `Error` objects unless it's a fatal system crash, as these are redacted to generic `500 Server Error` on the client for security.

```typescript
// Bad
if (user.role !== "admin") throw new Error("Unauthorized");

// Good
if (user.role !== "admin") {
  throw new ConvexError({ code: "UNAUTHORIZED", message: "Admin privileges required." });
}
```

## 6. Pagination & Data Fetching

### Pagination
For infinite scrolling or paginated tables, always use `paginate()`. It is cursor-based, ensuring UI consistency even if underlying data insertions/deletions occur.

```typescript
handler: async (ctx, args) => {
  return await ctx.db
    .query("logs")
    .withIndex("by_user", (q) => q.eq("userId", ctx.user._id))
    .order("desc") // newest first
    .paginate(args.paginationOpts);
}
```

### Eliminating Over-fetching
Do not blindly return entire database objects if the client does not need them. Map the results before returning them to limit payload sizes and obscure sensitive data.

## 7. HTTP Endpoints & Webhooks

When integrating with external providers (Stripe, Clerk, Twilio), youвЂ™ll need to receive webhooks. Do this in `convex/http.ts`.

- Keep HTTP handlers extremely fast.
- Verify webhook signatures using external libraries (e.g., `svix` for Clerk, `stripe` for Stripe).
- Delegate heavy processing side effects to an `internalAction` triggered via `ctx.runAction(...)` immediately after validation so you can return a `200 OK` back to the provider swiftly.

---

## 8. Background Jobs, Scheduling & Crons

### Scheduling Future Work
Use `ctx.scheduler.runAfter` or `ctx.scheduler.runAt` inside mutations/actions to schedule one-off tasks (e.g., sending an email 24 hours after a booking is confirmed). 

### Cron Jobs
Define all recurring jobs explicitly in `convex/crons.ts`.
- Keep the `cron` logic clean: the cron file should only orchestrate the timing, pointing to `internalAction`s or `internalMutation`s in domain folders.

```typescript
// convex/crons.ts
import { cronJobs } from "convex/server";
import { internal } from "./_generated/api";

const crons = cronJobs();

crons.daily(
  "Generate daily financial reports",
  { hourUTC: 0, minuteUTC: 0 },
  internal.reports.jobs.generateOverview
);

export default crons;
```

## 9. Rate Limiting

In enterprise apps, preventing abuse on public mutations (e.g., 'send SMS', 'login', 'create AI prompt') is vital.
Convex supports rate-limiting naturally through atomic mutations. Use `convex-helpers/server/rateLimit` to enforce sliding window or token bucket algorithms cleanly.

---

## 10. Migrations & Schema Evolution

Database schemas evolve. Convex allows hot-reloading code, but retrofitting existing documents requires data migrations.

- **Soft Defaulting**: Prefer adding optional fields to relations and handling `undefined` at the DB read layer over massive, immediate data migrations when possible.
- **Batched Migrations**: When a migration is unavoidable (e.g., splitting a table, recalculating aggregates), write a recursive mutation or an action that schedules batched internal mutations to process 1000 records at a time without hitting the mutation limit. Follow the pattern provided in `convex-helpers` migration utilities.

---

## 11. Environment Variables and Validation

Do not let bad config bring down production. Validate Environment Variables early.
Use a single file/function to ensure typesafety over `process.env`.

```typescript
// convex/env.ts
export function requireEnv(name: string): string {
  const value = process.env[name];
  if (!value) throw new Error(`Missing environment variable: ${name}`);
  return value;
}
```
